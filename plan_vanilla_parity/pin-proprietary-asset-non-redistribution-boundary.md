# Pin Proprietary Asset Non Redistribution Boundary

This document pins the proprietary asset non-redistribution boundary for the vanilla DOOM 1.9 parity rebuild. It freezes the canonical list of proprietary id Software assets that this repository must never bundle, regenerate, stage for commit, publish under any name, or redistribute in any form. It pins the gitignored, user-supplied drop locations under `doom/` and `iwad/` from which those assets may be loaded at runtime if (and only if) the user has placed their own purchased copy there. It pins the canonical list of permitted-with-notice GPL and utility files that may live alongside the proprietary assets in the same gitignored directory and that may be redistributed with their original notices intact. It pins the runtime evidence anchors in `src/reference/policy.ts` (`ASSET_BOUNDARIES`, `LicenseCategory`, `RedistributionPolicy`, `REFERENCE_BUNDLE_PATH`, `CODEX_WORKSPACE_PATH`) that the implementation already exposes so that downstream lanes can cross-check the boundary at test time. It pins the test-suite obligation: any test that exercises behavior tied to a proprietary asset must skip cleanly when the asset is absent rather than regenerate, fabricate, or substitute it. Any future change to these invariants must update this document and its focused test in lockstep.

## proprietary asset license category

The redistribution-forbidden category covers two distinct upstream license families. Files cataloged as `commercial-shareware` in `src/reference/policy.ts` (`DOOM1.WAD`, `DOOMD.EXE`, `DOOMDUPX.EXE`) are id Software shareware artifacts whose redistribution is restricted by the original DOOM shareware license. The single file cataloged as `mixed` in `src/reference/policy.ts` (`DOOM.EXE`) is a polyglot binary that contains both an id Software shareware DOS half and a GPL Chocolate Doom Windows half; the shareware half collapses the whole binary into the redistribution-forbidden bucket. The user-supplied registered and Ultimate `DOOM.WAD` (pinned by `plan_vanilla_parity/pin-user-supplied-registered-and-ultimate-iwad-scope.md`) is a proprietary id Software IWAD that is never redistributed even though it is not present in `src/reference/policy.ts` because no copy is bundled in the working tree.

## proprietary asset redistribution policy

The proprietary id Software assets pinned by this document must never be bundled, regenerated, staged for commit, published under any alternate name, or redistributed by this repository or by any artifact derived from it. They are user-supplied only. The repository must never ship a registered or Ultimate `DOOM.WAD` under any name, must never publish a copy of `DOOM.EXE`, `DOOMD.EXE`, `DOOMDUPX.EXE`, or `DOOM1.WAD`, and must never produce a derived binary or data file that embeds the bytes of any of these proprietary assets. Tests that exercise behavior tied to a proprietary asset must skip cleanly when the asset is absent at the documented user-supplied drop locations rather than regenerate, fabricate, or substitute it.

## user-supplied drop locations

- doom/
- iwad/

## user-supplied drop location rule

The two relative directories `doom/` and `iwad/` are gitignored in the repository's `.gitignore` and are reserved as user-supplied drop locations for the user's own purchased copies of the proprietary assets pinned by this document. The repository must never stage any file under either directory for commit. The repository may load the user's copy from those paths at runtime, but must never publish, copy, embed, or redistribute the loaded bytes anywhere outside those directories.

## permitted-with-notice filenames

- DOOMWUPX.exe
- chocolate-doom.cfg
- default.cfg
- smash.py

## permitted-with-notice rule

The four files listed above are not proprietary id Software assets. `DOOMWUPX.exe` is the UPX-packed Chocolate Doom Windows binary licensed under GPL v2 or later, `chocolate-doom.cfg` is the Chocolate Doom host configuration (GPL v2+), `default.cfg` is the vanilla DOOM defaults file generated by Chocolate Doom (GPL v2+), and `smash.py` is the binary merge utility distributed by the bundle author. These files may be redistributed when accompanied by their original license notices, but they share the gitignored `doom/` directory with the proprietary assets and so are not staged for commit by this repository either.
